Add the following functions to `apps/imports/re_services.py` (below the existing solicitor import code from Plan 01):

**1. `_build_header_mapping(fieldnames: list[str], alias_map: dict[str, list[str]]) -> dict[str, str]`** -- Internal helper:
   - Takes a list of actual CSV fieldnames and an alias mapping dict (canonical_name -> list of possible header names)
   - Returns a dict mapping canonical_name -> actual_fieldname found in the CSV
   - Case-insensitive matching: compare `header.strip().lower()` against aliases
   - This helper should be refactored from inline logic -- if Plan 01 already has similar logic for solicitor headers, extract to this shared helper and update solicitor import to use it too

**2. `merge_contact_fields(contact: Contact, new_data: dict) -> list[str]`** -- Merge-only update:
   - Takes an existing Contact and a dict of new field values
   - Only fills BLANK fields (empty string or None). Never overwrites existing non-blank values.
   - Fields to merge: `first_name`, `last_name`, `email`, `phone`, `phone_secondary`, `street_address`, `city`, `state`, `postal_code`, `country`, `organization_name`
   - For `email`: also skip if new_data value is empty
   - For `first_name`/`last_name`: also skip if new_data value is empty (never blank out names)
   - Returns list of field names that were actually updated
   - Does NOT call `contact.save()` -- caller saves with `update_fields`
   - Per CONTEXT.md: "Merge-only: only fill blank fields on the Contact. Never overwrite existing non-blank values."

**3. `_match_contact(row_data: dict, owner: User) -> tuple[Contact | None, str]`** -- Internal contact matching:
   - Implements three-tier match hierarchy per CONTEXT.md:
     a. `external_constituent_id` first: `Contact.objects.filter(external_constituent_id=ext_id).first()` (global, not owner-scoped, because external_constituent_id has a global unique constraint)
     b. If no ext_id match but email present: `Contact.objects.filter(owner=owner, email=email).first()` (owner-scoped per existing constraint)
     c. If no email match but phone present: `Contact.objects.filter(owner=owner, phone=phone).first()` (owner-scoped)
   - Returns `(contact, match_type)` where match_type is `'constituent_id'`, `'email'`, `'phone'`, or `'none'`
   - When ext_id matches but email/phone differs from existing Contact: log a warning (per CONTEXT.md: "trust the ID match but log a warning flagging the conflict")
   - The warning should include: row number, external_constituent_id, the mismatched field, old value, new value

**4. `import_re_constituents(file_bytes: bytes, filename: str, uploaded_by: User, owner: User) -> ImportBatch`** -- Orchestrator:
   - Step 1: SHA256 dedup check via `check_duplicate_import(file_bytes, ImportBatchType.RE_CONSTITUENT)`. If duplicate, return existing batch immediately.
   - Step 2: Decode with `decode_csv_bytes(file_bytes)`.
   - Step 3: Parse with `csv.DictReader`. Build header mapping with aliases:
     - `constituent_id`: `['cnbio_id', 'consid', 'constituent_id', 'cons_id', 'id']`
     - `first_name`: `['cnbio_first_name', 'firstname', 'first_name', 'first name', 'fname']`
     - `last_name`: `['cnbio_last_name', 'lastname', 'last_name', 'last name', 'lname']`
     - `organization_name`: `['cnbio_org_name', 'orgname', 'org_name', 'organization', 'organization_name', 'org name']`
     - `email`: `['cnadrprf_email', 'email', 'email_address', 'email address', 'emailaddress']`
     - `phone`: `['cnph_1_01_phone_number', 'phone', 'phone_number', 'phone number', 'phonenumber']`
     - `street_address`: `['cnadrprf_addrline1', 'address', 'address_line_1', 'address line 1', 'street', 'street_address']`
     - `city`: `['cnadrprf_city', 'city']`
     - `state`: `['cnadrprf_state', 'state']`
     - `postal_code`: `['cnadrprf_zip', 'zip', 'postal_code', 'postal code', 'zipcode', 'zip_code']`
     - `country`: `['cnadrprf_contrylongdsc', 'country']`
   - Validate: at least one of `first_name`, `last_name`, or `organization_name` must be a mapped header (file-level error if none found).
   - Step 4: Iterate rows with error collection:
     - Track `row_number` (starting at 2 for header offset).
     - Collect errors as `[{"row": int, "error": str}]`.
     - **Minimum data validation** (per Claude's discretion in CONTEXT.md): Require at least one of: (a) `first_name` + `last_name` both non-empty, or (b) `organization_name` non-empty. Rows with none of these are errors: "Row N: Missing name or organization -- cannot create contact".
     - For each valid row:
       a. Build `row_data` dict mapping canonical field names to stripped CSV values.
       b. Set `external_constituent_id` from the CSV if present.
       c. Call `_match_contact(row_data, owner)`.
       d. If match found:
          - If matched by `constituent_id` and email/phone differs: log warning (already handled in `_match_contact`).
          - Call `merge_contact_fields(contact, row_data)` to fill blank fields.
          - If any fields were updated: `contact.save(update_fields=updated_fields)`, increment `updated_count`.
          - If no fields updated: increment `skipped_count`.
          - If contact doesn't have `external_constituent_id` but CSV row does: set it and include in save.
       e. If no match:
          - Create new Contact with: `owner=owner`, all available fields from row_data, `external_constituent_id` from CSV.
          - `status` defaults to ContactStatus.PROSPECT (model default).
          - Increment `created_count`.
       f. Catch per-row exceptions (IntegrityError, ValidationError, etc.) as row errors -- never abort.
   - Step 5: Create ImportBatch record with:
     - `import_type=ImportBatchType.RE_CONSTITUENT`
     - `status=ImportBatchStatus.COMPLETED` (or FAILED for file-level errors)
     - `sha256_hash`, `filename`, `uploaded_by`
     - `total_rows`, `created_count`, `updated_count`, `skipped_count`, `error_count`
     - `summary` JSON: `{"errors": [...], "warnings": [...]}`
   - Wrap import processing in `transaction.atomic()`. If file-level error, create batch with FAILED status outside the transaction.

**Step 1: Create `apps/imports/management/commands/import_re_constituents.py`**

Django management command following the same pattern as `import_re_solicitors` from Plan 01:
- `help = 'Import RE Constituent CSV file'`
- Arguments:
  - `file` (positional): Path to CSV file
  - `--owner` (required): Email of the user who will own new contacts AND who is recorded as `uploaded_by`
- `handle()` method:
  1. Look up owner user by email. Raise `CommandError` if not found or user is not active.
  2. Read file bytes. Raise `CommandError` if file not found.
  3. Check file size <= 10 MB. Raise `CommandError` if too large.
  4. Print `f'Processing {filename}...'`
  5. Call `import_re_constituents(file_bytes=file_bytes, filename=os.path.basename(file_path), uploaded_by=owner_user, owner=owner_user)`.
  6. Check if batch is duplicate: print `self.style.WARNING('File already imported...')` with batch info and return.
  7. Check if batch failed: print `self.style.ERROR(f'Import failed: {batch.summary.get("file_error", "Unknown error")}')` and return.
  8. Print summary:
     - `self.style.SUCCESS(f'Import complete: {batch.filename}')`
     - `f'  Created: {batch.created_count}'`
     - `f'  Updated: {batch.updated_count}'`
     - `f'  Skipped: {batch.skipped_count}'`
     - `f'  Errors: {batch.error_count}'`
     - If warnings in summary: print count `f'  Warnings: {len(warnings)}'`
     - If errors exist: print first 10 error messages with row numbers
