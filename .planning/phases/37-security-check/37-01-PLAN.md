  <action>
    **Fix the refresh token rotation bug** identified in research:

    The issue is at line 106 of `frontend/src/api/client.ts`. The token refresh interceptor currently does:
    ```typescript
    const { access } = response.data
    localStorage.setItem(ACCESS_TOKEN_KEY, access)
    ```

    It only stores the new access token, ignoring the new refresh token returned by the server. Since `ROTATE_REFRESH_TOKENS=True` and `BLACKLIST_AFTER_ROTATION=True` in Django settings, the old refresh token is blacklisted after use. The next time the access token expires, the frontend tries to use the now-blacklisted old refresh token, which fails, logging the user out.

    **Fix:**
    1. Destructure both `access` and `refresh` from the response data:
       ```typescript
       const { access, refresh } = response.data
       ```
    2. Use the existing `setTokens()` helper function (which stores both tokens) instead of manually setting just the access token:
       ```typescript
       setTokens(access, refresh)
       ```
    3. Verify that `setTokens` is already defined in the same file (it should be -- it stores both ACCESS_TOKEN_KEY and REFRESH_TOKEN_KEY in localStorage).

    **Important:** Do NOT change the error handling or the request queue logic. Only change the token storage after a successful refresh.
  </action>
