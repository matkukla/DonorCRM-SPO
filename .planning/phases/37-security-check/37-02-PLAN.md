  <action>
    **Install django-csp:**

    1. Add `django-csp>=4.0,<5.0` to `requirements/base.txt` (production dependency, not just dev).

    **Configure CSP in prod.py:**

    2. Add `'csp.middleware.CSPMiddleware'` to `MIDDLEWARE` in `config/settings/prod.py`. Insert it after Django's `SecurityMiddleware` (position 1 or 2 in the list). Check how MIDDLEWARE is defined in prod.py -- it may extend base.py's list.

    3. Configure CSP using django-csp 4.0's `CONTENT_SECURITY_POLICY` dict format:
       ```python
       CONTENT_SECURITY_POLICY = {
           "EXCLUDE_URL_PREFIXES": ["/admin", "/api/v1/docs", "/api/v1/redoc"],
           "DIRECTIVES": {
               "default-src": ["'none'"],
               "script-src": ["'self'"],
               "style-src": ["'self'"],
               "img-src": ["'self'"],
               "connect-src": ["'self'"],
               "frame-ancestors": ["'none'"],
               "form-action": ["'self'"],
               "base-uri": ["'self'"],
           },
       }
       ```

       **Key design decisions:**
       - Use `EXCLUDE_URL_PREFIXES` to skip `/admin`, `/api/v1/docs/`, and `/api/v1/redoc/` because Django admin and Swagger UI use inline styles/scripts.
       - The CSP is strict (`default-src: 'none'`) because Django serves only JSON API responses -- no HTML pages need scripts/styles.
       - `frame-ancestors: 'none'` reinforces the existing `X-FRAME-OPTIONS = 'DENY'`.

    **Add Referrer-Policy and Permissions-Policy:**

    4. In `config/settings/prod.py`, add:
       ```python
       SECURE_REFERRER_POLICY = 'strict-origin-when-cross-origin'
       ```
       This is supported natively by Django's SecurityMiddleware (Django 4.2+).

    5. Since Django does not have built-in `Permissions-Policy` support, add a simple middleware or use a response header in the SecurityMiddleware. The simplest approach: Add a small middleware class in `config/settings/prod.py` (or a separate file) that sets the `Permissions-Policy` header. Alternatively, if the existing middleware stack supports it, add the header there.

       Actually, the simplest approach is to add `Permissions-Policy` via render.yaml headers on the backend service (if supported by Render for web services). If not, create a tiny middleware. For now, focus on the frontend static site headers in Task 2, and set `Permissions-Policy` there. For the Django API, skip Permissions-Policy (it's primarily a browser-enforced policy for HTML pages, and Django serves JSON). Document this decision.

    **Run pip install:**

    6. Run `pip install django-csp>=4.0,<5.0` to install the dependency.

  <name>Task 2: Restrict API docs to authenticated users and add frontend security headers</name>
  <files>
    config/api_urls.py

  <action>
    **Restrict API documentation in production:**

    1. In `config/api_urls.py`, wrap the schema/docs/redoc views with authentication.

       Option A (preferred -- conditional based on DEBUG): Import `settings` and conditionally apply permissions:
       ```python
       from django.conf import settings
       from rest_framework.permissions import IsAuthenticated
       from drf_spectacular.views import SpectacularAPIView, SpectacularSwaggerView, SpectacularRedocView

       # In production, require authentication for API docs
       schema_permission = [IsAuthenticated] if not settings.DEBUG else []

       urlpatterns = [
           # ...
           path('schema/', SpectacularAPIView.as_view(permission_classes=schema_permission), name='schema'),
           path('docs/', SpectacularSwaggerView.as_view(url_name='schema', permission_classes=schema_permission), name='swagger-ui'),
           path('redoc/', SpectacularRedocView.as_view(url_name='schema', permission_classes=schema_permission), name='redoc'),
           # ...
       ]
       ```

       Option B (alternative): Use drf-spectacular settings to set default permissions. Check if `SPECTACULAR_SETTINGS` already exists in settings.

       Go with Option A -- it's explicit and keeps docs open in dev for convenience.

    **Add frontend static site security headers in render.yaml:**

    2. In `render.yaml`, find the frontend static site configuration and add security headers. Render static sites support a `headers` section:
       ```yaml
       headers:
         - path: /*
           name: Content-Security-Policy
           value: "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self' https://*.onrender.com; frame-ancestors 'none'; form-action 'self'; base-uri 'self'"
         - path: /*
           name: X-Content-Type-Options
           value: nosniff
         - path: /*
           name: Referrer-Policy
           value: strict-origin-when-cross-origin
         - path: /*
           name: Permissions-Policy
           value: "camera=(), microphone=(), geolocation=(), payment=()"
       ```

       **CSP details for the React SPA:**
       - `'unsafe-inline'` in `style-src` is needed because many UI libraries (shadcn/ui, Tailwind) inject inline styles
       - `connect-src` must include the backend API origin (`https://*.onrender.com` or the specific backend URL if available from render.yaml)
       - `img-src 'self' data:` allows data URIs used by some UI components
       - Read the existing render.yaml to check the backend URL and existing headers config

       **Important:** Check the existing `headers` section in render.yaml. The research mentions `Cache-Control: no-cache` is already set. Merge with existing headers, don't replace them.
