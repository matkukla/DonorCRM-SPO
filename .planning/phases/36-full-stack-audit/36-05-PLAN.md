  <action>
Sweep all frontend pages and components for missing dark mode classes. Research identified these high-risk areas:

**Methodology:**
1. Search for hardcoded light-mode-only colors that lack dark: variants:
   ```bash
   grep -rn "bg-white\|bg-gray-50\|bg-gray-100\|text-gray-900\|text-gray-800\|text-gray-700\|border-gray-200\|border-gray-300" frontend/src/ --include="*.tsx" --include="*.ts" | grep -v "dark:" | grep -v "node_modules"
   ```
   Each result is a potential dark mode gap. For each:
   - `bg-white` -> add `dark:bg-gray-900` or `dark:bg-background` (prefer semantic tokens)
   - `bg-gray-50` -> add `dark:bg-gray-900/50` or `dark:bg-muted`
   - `bg-gray-100` -> add `dark:bg-gray-800`
   - `text-gray-900` -> add `dark:text-gray-100` or `dark:text-foreground`
   - `text-gray-800` -> add `dark:text-gray-200`
   - `text-gray-700` -> add `dark:text-gray-300`
   - `border-gray-200` -> add `dark:border-gray-700` or `dark:border-border`
   - `border-gray-300` -> add `dark:border-gray-600`

2. **Priority pages** (v2.0 additions most likely to have gaps):
   - Prayer focus mode (custom amber chapel styling -- needs dark equivalents)
   - Import result banners (success/error/duplicate state styling)
   - Admin analytics pages (charts, heatmaps)
   - Any custom styled components that don't use shadcn/ui defaults

3. Follow the established pattern from Phase 21 decisions:
   - Status backgrounds: `bg-*-50 dark:bg-*-950/50`
   - Status text: `text-*-600 dark:text-*-400` for icons, `text-*-800 dark:text-*-200` for body
   - Status borders: `border-*-200 dark:border-*-800`
   - Prefer semantic tokens (bg-background, text-foreground, border-border) over hardcoded grays

4. For recharts components in admin analytics: Check if chart colors use CSS variables that respond to dark mode. If hardcoded hex colors, convert to CSS variables or use the theme-aware palette.

  <action>
Audit accessibility across the frontend. Focus on three areas:

**A. ARIA labels on icon-only buttons:**
1. Find all icon-only buttons (buttons with no text content, only an icon):
   ```bash
   grep -rn "<Button.*variant.*size.*icon\|<button.*onClick" frontend/src/ --include="*.tsx" | head -50
   ```
2. For each icon-only button, add `aria-label="descriptive text"`. Examples:
   - Delete button with trash icon: `aria-label="Delete"`
   - Edit button with pencil icon: `aria-label="Edit"`
   - Close button with X icon: `aria-label="Close"`
   - Sidebar toggle: `aria-label="Toggle sidebar"`
   - Theme toggle: `aria-label="Toggle theme"`
3. Also check icon-only links (`<a>` or `<Link>` with only an icon).

**B. Table accessibility:**
1. Find all table components (both DataTable and plain HTML tables):
   ```bash
   grep -rn "<table\|<Table\|DataTable" frontend/src/ --include="*.tsx" | head -30
   ```
2. For each table:
   - Add `aria-label` describing the table content (e.g., `aria-label="Contacts list"`)
   - Ensure `<th>` elements have `scope="col"` for column headers
   - The prayer intentions plain HTML table (Phase 33) specifically needs these attributes

**C. Focus indicators:**
1. Check if Tailwind's default focus ring is enabled. Look for `focus:ring` or `focus-visible:ring` patterns.
2. If the project uses `outline-none` broadly (sometimes done in reset CSS), ensure `focus-visible:ring-2 focus-visible:ring-ring` is applied to interactive elements.
3. Verify that custom interactive elements (drag handles, prayer focus mode keyboard nav, sidebar items) have visible focus states.

**D. Document remaining accessibility items:**
For items that need live browser verification (actual contrast ratios, screen reader testing), document them as items for the human testing checklist in Plan 06 rather than trying to fix blindly.
