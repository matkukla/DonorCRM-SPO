Research found inconsistent error response formats across the API:

**Error response unification:**

1. **Find all manual Response({'error': ...}) calls:**
   ```bash
   grep -rn "Response({'error'" apps/ --include="*.py"
   grep -rn "Response({\"error\"" apps/ --include="*.py"
   ```
   DRF convention is `{'detail': '...'}`. Change all `{'error': '...'}` responses to `{'detail': '...'}`. Research says this affects ~8 occurrences primarily in insights views.

2. **Check for other non-standard error patterns:**
   ```bash
   grep -rn "Response({" apps/ --include="*.py" | grep -v "status=200\|status=201\|status=204"
   ```
   Ensure all error responses use `{'detail': '...' }` format consistently.

**Inline serializer cleanup:**

3. **FundListSerializer in imports/views.py**: Research says this is defined inline instead of in a serializers module. Read the file to find it. If it's a simple serializer (< 10 lines), it's acceptable to leave inline with a comment. If it's more complex, move it to `apps/imports/serializers.py`.

4. **ImportBatchListView hand-built dicts**: Research says this view builds response dicts manually instead of using a serializer. This was a conscious Phase 32 decision ("No serializer class -- hand-built dict for simplicity, only 12 fields"). Leave this as-is per the locked decision, but add a comment explaining why.

**Owner scoping consistency audit:**

5. Review the owner scoping logic across views. Research identified two patterns:
   - `user.role not in ['admin', 'finance', 'read_only']` (multi-role check)
   - `user.role == 'admin'` (single role check)
   Document the actual access control matrix in a comment at the top of `apps/core/permissions.py`:

   ```python
   # read_only: read access to own resources only
   # staff: full access to own resources only
   ```
   Verify this matrix matches what the code actually implements. If any view's scoping doesn't match this matrix, fix it.
