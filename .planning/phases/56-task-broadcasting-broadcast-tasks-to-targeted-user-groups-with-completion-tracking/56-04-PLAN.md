Create frontend/src/pages/tasks/BroadcastTaskDialog.tsx:

This is a Dialog component with two states: form entry and confirmation.

Props interface:
```typescript
interface BroadcastTaskDialogProps {
  open: boolean
  onOpenChange: (open: boolean) => void
}
```

Component structure:

**State:**
- `step`: "form" | "confirm" (default "form")
- `title`: string (required)
- `description`: string
- `taskType`: TaskType (default "other")
- `priority`: TaskPriority (default "medium")
- `dueDate`: string (required)
- `targetType`: BroadcastTargetType (default based on role: admin -> "all_missionaries", supervisor -> "my_team")
- `specificUserIds`: string[] (for specific_users target)
- `recipientCount`: number (computed from target selection)

**Form step:**
Uses Dialog with `sm:max-w-lg` className on DialogContent.

DialogHeader:
- DialogTitle: "Broadcast Task"
- DialogDescription: "Create a task for multiple users at once."

Form fields (use Label + Input/Select pattern from existing forms):
1. Title: `<Input>` (required, max 255 chars)
2. Description: `<Textarea>` (optional)
3. Type: `<Select>` with TaskType options using taskTypeLabels
4. Priority: `<Select>` with TaskPriority options using taskPriorityLabels
5. Due Date: `<Input type="date">` (required)
6. Target: Radio group:
   - For admin: "All Missionaries", "All Supervisors", "Specific Users"
   - For supervisor: "My Team", "Specific Members" (maps to "specific_users" target_type)
   - Use `<RadioGroup>` from @radix-ui/react-radio-group or simple buttons with active state styling
7. Specific Users (shown only when targetType == "specific_users"):
   - Checkbox list of users
   - Admin: fetch viewable users from /api/v1/users/viewable/ (lazy approach: just show a multi-select or use user.supervised_users for supervisor)
   - Supervisor: show user.supervised_users from AuthProvider
   - Each user: `<Checkbox>` + name label
   - Selected users stored in specificUserIds array

Compute recipientCount:
- "all_missionaries" / "all_supervisors": show "all" (exact count shown after API call in confirmation)
- "my_team": user.supervised_users?.length || 0
- "specific_users": specificUserIds.length

DialogFooter: Cancel button + "Next" button (disabled if title or dueDate empty, or if specific_users selected but none chosen)

**Confirmation step:**
DialogHeader: "Confirm Broadcast"
Content: Summary of task details (title, due date, priority, target) + message:
"This will create a task for {recipientCount} users. Proceed?"

For "all_missionaries"/"all_supervisors" where exact count isn't known client-side, show "This will create a task for all {target label}. Proceed?"

DialogFooter: "Back" button (returns to form step) + "Send Broadcast" button (calls mutation)

**Mutation:**
Use `useCreateBroadcast()` hook. On success:
- Toast: `toast.success("Broadcast sent to {count} users")` using sonner
- Reset form state
- Close dialog (onOpenChange(false))
- Reset step to "form"

On error: `toast.error("Failed to create broadcast")`

Important implementation details:
- Import `Megaphone` icon from lucide-react for the dialog title decoration
- Import `toast` from `sonner`
- Use `useAuth()` to get user role and supervised_users
- When dialog closes (onOpenChange), reset step to "form"
- For the specific users multi-select for admin role: use a simple approach -- admin creates with specific_user_ids if "Specific Users" is selected, OR for the admin "All Missionaries"/"All Supervisors" targets, specific_user_ids is omitted. The backend resolves the full list.
- For supervisor viewing their team: pull from `user.supervised_users` which is already available in AuthProvider context
