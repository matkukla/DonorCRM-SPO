
FIX — read directly from user object (API now returns these arrays):
```typescript
// Read from user.supervisor_ids and user.coach_ids (from UserSerializer M2M arrays):
setEditSupervisedUserIds(user.supervisor_ids ?? [])
setEditCoachedUserIds(user.coach_ids ?? [])
```

Note: user.supervisor_ids here is in the context of "which missionaries does THIS supervisor oversee."
Wait — this is the SUPERVISOR user's edit form. The supervisor's user object from UserSerializer now has:
- supervisor_ids: which supervisors are assigned TO this user (if they're a missionary)
- coach_ids: which coaches are assigned TO this user (if they're a missionary)

But for supervisors, the edit form needs to set WHICH MISSIONARIES they supervise (supervised_user_ids in UserUpdate).
The UserSerializer now returns supervised_users via CurrentUserSerializer, but UserSerializer (admin list) returns supervisor_ids/coach_ids on the user's own assignments.

The correct approach: The UserSerializer returns supervisor_ids[] meaning "this missionary's supervisors".
But openEditDialog() for a SUPERVISOR user needs to know which missionaries that supervisor oversees.

After M2M, the UserSerializer does NOT return a "my_missionaries" list for supervisor users. The supervised_users reverse M2M is not exposed in UserSerializer.

SOLUTION: Add a `supervised_missionary_ids` SerializerMethodField to UserSerializer (backend, Plan 03 already done or amend 46-03 to include it). OR derive it in the frontend by looking at ALL missionaries and checking which ones include this supervisor's ID in their supervisor_ids array.

Frontend derivation (simpler, no backend change):
```typescript
// users = all users from useUsers()
// After M2M, each missionary user has supervisor_ids: string[]
// So find missionaries supervised by this supervisor:
const supervised = users?.filter(u =>
  u.role === 'missionary' && u.supervisor_ids?.includes(user.id)
).map(u => u.id) ?? []
setEditSupervisedUserIds(supervised)
```

This works because the UserSerializer now returns supervisor_ids[] for each user (the supervisors of that missionary), so filtering missionaries by `u.supervisor_ids.includes(supervisorUser.id)` correctly returns all missionaries assigned to that supervisor.

Similarly for coaches:
```typescript
const coached = users?.filter(u =>
  u.role === 'missionary' && u.coach_ids?.includes(user.id)
).map(u => u.id) ?? []
setEditCoachedUserIds(coached)
```
